##  Architecture & Core Concepts
To ensure enterprise-grade code quality, several intentional design choices were made throughout the application:

### 1. Strict Constructor Injection
**Implementation:** Zero field injection (`@Autowired`) or setter injection was used.
**Why:** Constructor injection guarantees that required dependencies are available at initialization, prevents `NullPointerException`s, allows fields to be marked as `final` for immutability, and makes unit testing significantly easier.

### 2. Thread-Safe In-Memory Data Storage
**Implementation:** The `UserRepository` utilizes a `CopyOnWriteArrayList` and an `AtomicLong` instead of a standard `ArrayList` and primitive `long`.
**Why:** Standard collections are not thread-safe. In a web environment where multiple HTTP requests (threads) might attempt to create a user simultaneously, `CopyOnWriteArrayList` prevents `ConcurrentModificationException`s, and `AtomicLong` ensures no two users ever receive the exact same ID.

### 3. Defensive Copying
**Implementation:** The `findAll()` method returns `new ArrayList<>(users)`.
**Why:** This prevents internal repository state from leaking. If a higher layer (like a Service) mutates the returned list, the actual master data inside the Repository remains uncorrupted.

### 4. Dynamic Bean Injection (Strategy Pattern)
**Implementation:** For the Dynamic Message Formatter (Task C), multiple beans implementing the `MessageFormatter` interface were created. The `MessageService` injects a `List<MessageFormatter>` via the constructor.
**Why:** This completely eliminated the need for `if-else` or `switch` statements in the controller/service to determine the message format. Spring automatically injects all active implementations, allowing the service to dynamically filter and select the correct bean at runtime based on the request parameter.

### 5. Global Exception Handling
**Implementation:** Implemented a `@RestControllerAdvice` class.
**Why:** Instead of allowing the application to crash or return messy HTML stack traces on errors (like a missing user ID), the handler catches `RuntimeException` and `IllegalArgumentException`, formatting them into clean, standardized JSON responses.

##  API Endpoints & Testing Done Using:

### Task A: User Management System
* **GET All Users**
    * `GET http://localhost:8080/users`
    * *Returns a JSON array of all registered users.*
* **GET User by ID**
    * `GET http://localhost:8080/users/{id}`
    * *Returns a specific user or a 404 error if not found.*
* **CREATE User**
    * `POST http://localhost:8080/users`
    * *Body (JSON):*
      ```json
      {
        "name": "John Doe",
        "email": "john@example.com"
      }
      ```

### Task B: Notification System
* **Trigger Notification**
    * `POST http://localhost:8080/notifications/trigger`
    * *Returns a simple success string generated by a reusable `@Component`.*

### Task C: Dynamic Message Formatter
* **Get Short Message**
    * `GET http://localhost:8080/message?type=SHORT`
    * *Dynamically routes to the `ShortMessageFormatter` component.*
* **Get Long Message**
    * `GET http://localhost:8080/message?type=LONG`
    * *Dynamically routes to the `LongMessageFormatter` component.*
